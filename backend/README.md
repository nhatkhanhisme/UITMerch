# UITMerch Backend

Spring Boot 3.3 / Java 21 REST API for the UIT merchandise platform.
Multi-role e-commerce supporting Customers, Organizers, and Admins.

---

## Quick Start

### Option 1 ÔÇö Docker (recommended for frontend testing)

```bash
cd backend
docker compose up --build
```

- API Ôćĺ `http://localhost:8080`
- Swagger UI Ôćĺ `http://localhost:8080/swagger-ui.html`

No `.env` file required. Sample data is seeded automatically on first startup and persists across restarts.

OTPs are not emailed ÔÇö fetch them directly:
```
GET http://localhost:8080/api/v1/dev/otps?email=<email>
```

```bash
docker compose up           # subsequent starts (faster, uses cached image)
docker compose down         # stop (data volume kept)
docker compose down -v      # stop + wipe DB (forces re-seed on next up)
docker compose logs -f backend
```

### Option 2 ÔÇö Dev profile (local, H2 in-memory)

```bash
./mvnw spring-boot:run -Dspring-boot.run.profiles=dev
# Windows: .\mvnw.cmd spring-boot:run -Dspring-boot.run.profiles=dev
```

No env vars required. Schema is generated by Hibernate. Data resets on every restart.

OTPs are not emailed ÔÇö fetch them directly:
```
GET http://localhost:8080/api/v1/dev/otps?email=<email>
```

H2 console (for DB inspection):
```
http://localhost:8080/h2-console
JDBC URL: jdbc:h2:mem:uitmerch  |  User: sa  |  Password: (empty)
```

### Option 3 ÔÇö Production (PostgreSQL + Supabase)

```bash
cp .env.example .env
# fill in values ÔÇö see Environment Variables section below
./mvnw spring-boot:run
# Windows: .\mvnw.cmd spring-boot:run
```

## Tech Stack

| Layer | Technology |
|---|---|
| Framework | Spring Boot 3.3.5, Java 21 |
| Database | PostgreSQL 16, Flyway (V1ÔÇôV28 migrations) |
| Auth | JWT via JJWT 0.12.x ÔÇö stateless, access + refresh tokens, type-checked |
| Token blacklist | PostgreSQL-backed `invalidated_tokens` table ÔÇö survives restarts |
| Rate limiting | In-memory sliding-window (`RateLimiterService`) ÔÇö login, register, OTP, guest checkout |
| Storage | Supabase Storage (S3-compatible via AWS SDK) |
| Email | JavaMail (SMTP) with `@Async` dispatch ÔÇö OTP, password reset, order status |
| Cache | Spring `ConcurrentMapCache` ÔÇö categories, popular merch |
| API Docs | springdoc-openapi 2.6 ÔÇö Swagger UI at `/swagger-ui.html` (disabled by default in prod) |
| Tests | 169 tests across 14 test classes ÔÇö unit (Mockito) + `@DataJpaTest` integration |
| Logging | Logback ÔÇö human-readable (dev/docker), structured JSON via logstash-logback-encoder (prod) |

## API Reference

### Auth ÔÇö `/api/v1/auth`

| Method | Path | Auth | Description |
|---|---|---|---|
| POST | `/register` | None | Register a customer account ÔÇö sends OTP |
| POST | `/register/organizer` | None | Register an organizer account ÔÇö sends OTP |
| POST | `/verify-email` | None | Verify email with OTP (5 attempts max; 15-min lockout) |
| POST | `/resend-otp` | None | Re-issue OTP for unverified accounts (silent no-op if email unknown) |
| POST | `/login` | None | Login ÔÇö returns `token` (access) + `refreshToken` |
| POST | `/refresh` | None | Exchange refresh token for new access + refresh token pair (rotated) |
| POST | `/logout` | Bearer | Blacklist the current JWT ÔÇö persisted across restarts |
| POST | `/forgot-password` | None | Send password-reset OTP (silent no-op if email unknown) |
| POST | `/reset-password` | None | Reset password using OTP |

**Rate limits (per IP):** login 10/15 min ┬Ě register & resend-otp & forgot-password 5/hr ┬Ě guest checkout 20/hr

### Cart ÔÇö `/api/v1/customer/cart` *(CUSTOMER)*

### Orders ÔÇö Customer `/api/v1/customer/orders` *(CUSTOMER)*

| Method | Path | Description |
|---|---|---|
| GET | `/` | List own orders (`?status=` filter, paginated) |
| GET | `/{id}` | Get order by ID |
| POST | `/instant` | Instant order ÔÇö single item without going through cart |
| PATCH | `/{id}/cancel` | Cancel a PENDING order (body: `cancelReason`, optional `cancelReasonNote`) ÔÇö stock restored |

### Orders ÔÇö Organizer `/api/v1/organizations/{orgId}/orders` *(ORGANIZER)*

| Method | Path | Description |
|---|---|---|
| GET | `/orders` | List org orders (`?status=` filter, paginated) |
| GET | `/orders/{id}` | Get order by ID |
| PATCH | `/orders/{id}/status` | Advance order status (CONFIRMED Ôćĺ READY for single order) |
| PATCH | `/orders/{id}/checkin` | Mark READY order as COMPLETED (campus pickup check-in) |
| PATCH | `/orders/{id}/cancel` | Cancel PENDING or CONFIRMED order (body: `cancelReason`, optional `cancelReasonNote`) ÔÇö stock restored |
| POST | `/pickup-schedules` | Create a pickup schedule for a batch of CONFIRMED orders ÔÇö transitions them all to READY and sends customer notifications |
| GET | `/pickup-schedules` | List org pickup schedules (paginated) |
| GET | `/pickup-schedules/{scheduleId}/orders` | List orders assigned to a pickup schedule (check-in list) |

**Order status transitions:**

```
PENDING Ôćĺ CONFIRMED Ôćĺ READY Ôćĺ COMPLETED
PENDING Ôćĺ CANCELLED                          (stock restored)
CONFIRMED Ôćĺ CANCELLED                        (stock restored)
```

`READY` is reached via `createPickupSchedule` (batch) or `updateOrderStatus` (single). `COMPLETED` is set by `checkInOrder` when the customer physically collects at campus. An email notification is sent to the customer on every transition; a push notification is also created in-app when the order reaches `READY`.

## API Authentication

1. `POST /api/v1/auth/login` Ôćĺ copy `data.token`
2. Swagger UI Ôćĺ click **Authorize** Ôćĺ paste token Ôćĺ **Authorize**
3. All protected endpoints now work

> Swagger is disabled in production by default. Enable it with `SWAGGER_ENABLED=true`.

All responses follow this envelope:
```json
{
  "success": true,
  "message": "Human-readable string",
  "data": { },
  "meta": null,
  "traceId": "uuid"
}
```

Paginated `meta`:
```json
{ "page": 0, "size": 20, "total": 100 }
```

Error responses: `success: false`, `message` explains what went wrong, `data: null` (or field map for validation errors).

| `APP_JWT_EXPIRATION` | `86400000` | Access token TTL (ms) ÔÇö 24 h |
| `APP_JWT_REFRESH_EXPIRATION` | `604800000` | Refresh token TTL (ms) ÔÇö 7 days |
